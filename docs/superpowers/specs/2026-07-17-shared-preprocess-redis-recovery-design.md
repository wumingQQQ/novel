# 小说级公共预处理 Redis 恢复设计

## 背景

章节切分和 Passage 构建是同一小说的公共产物，但现有失败检查点以 `jobId` 保存在 Redis。两个角色任务处理同一本小说时，后到任务无法接管前一任务的失败章节；已经写入的 Passage 还可能被后续任务当作未完成产物清理并重建。

此外，SSE 客户端断连在当前实现中可能向业务流水线传播 `IllegalStateException`，使已成功落库的章节被记录为失败。

## 目标

- 公共阶段的生产权、进度、章节级成功/失败项和重试预算均按 `(novelId, stage)` 共享。
- 任一角色任务到达公共阶段时，自动由小说级锁选出唯一生产者；生产者只重试失败章节。
- 未取得锁的任务等待共享阶段结果，成功后复用，失败后自动参与下一轮锁竞争。
- SSE 断连只影响订阅，不影响章节构建、公共阶段状态或任务结果。
- 共享阶段每个 `(novelId, stage)` 最多自动尝试三次；耗尽后为终态失败，保留失败原因。

## 非目标

- 本次不修改数据库 schema。
- 角色样本、反应规则和画像等角色专属阶段仍按 `jobId` 维护检查点。
- 本次不实现跨 Redis 数据丢失后的章节级恢复；该能力留给持久化检查点方案。

## 设计

### Redis 检查点

为公共阶段引入独立存储，键按小说和阶段命名：

- `novel:{novelId}:{stage}:success`：成功章节 ID。
- `novel:{novelId}:{stage}:failed`：失败章节 ID。
- `novel:{novelId}:{stage}:attempts`：共享自动尝试次数。

首次执行且没有失败项时处理该小说阶段的全部章节；后续生产者消费失败列表，仅处理失败章节。成功时清理三个检查点键；失败时保留失败项和尝试次数，供下一持锁任务接管。

### 锁与状态转换

`NovelPreprocessLock` 继续使用 `(novelId)` 作为互斥范围。只有锁持有者执行公共工作；等待者轮询 `novel_preprocesses` 的状态，看到 `READY` 后复用。阶段失败后，未超过共享预算的等待者可再次取得锁并恢复失败项。预算耗尽时，协调器保留 `FAILED` 状态和失败原因，不再自动执行。

`novel_preprocesses` 仍是公共阶段整体状态源：`READY` 表示所有章节完成，`FAILED` 表示共享预算耗尽或当前恢复未成功。Redis 只保存章节级临时检查点。

### SSE 隔离

SSE 推送是尽力而为的旁路。`SseEmitter.send` 失败时移除订阅；后续关闭 emitter 的异常必须被吞掉，或不再显式关闭已出错的 emitter。任何进度推送异常都不得传播到 `PassageBuildStep` 或公共预处理协调器。

## 验收

1. 两个角色任务同时到达同一本小说的 Passage 阶段时，只有一个执行章节切分；另一个等待后复用。
2. 第一轮仅部分章节失败时，下一位锁持有者只处理失败章节，不删除成功章节的 Passage。
3. SSE 订阅断开后，Passage 已成功的章节仍被标记成功，公共阶段可进入 `READY`。
4. 同一公共阶段自动失败三次后停止自动接管，并保留统一失败信息。
5. README 说明未来将以持久化阶段子项表替代 Redis 章节检查点。

## 后续方向

新增持久化阶段子项表，以 `(novel_id, stage, item_id)` 唯一标识章节状态，保存状态、重试次数和失败原因。届时它替代 Redis 作为章节级恢复与审计的事实来源，Redis 锁仅保留调度职责。
